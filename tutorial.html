Endev Tutorial

The goal of the tutorial is to get familiar with Endev and how to use it for rapid prototyping. To achieve this you will:

  • Learn about Endev on-the-fly creation of own data
  • Learn about the real-time nature of Endev’s own data and storing it in the cloud
  • Learn about accessing API data (through YQL or REST)
  • Combine data from different data sources using Endev
  • Provide feedback on Endev’s design

0. Rapid prototyping, no JavaScript, declarative annotations and data

The purpose of Endev is to empower non-programmers and programmers alike to build interactive interfaces that allow you to display and edit data. And to do it fast. That way you can quickly build a demo of your idea to test with the potential users or clients, or build an interface for your personal birthday wish list so that even your grandmother can use it :)

There are certain challenges we are faced with when working with data. From where you store it, how you read the data that’s all around (more than 14k APIs out there) to the the technical challenges (spaghetti-code, endless callbacks, etc.).

Endev address these through two core principles:

First, declarative annotations hide the nitty-gritty aspects of writing code (that’s often the same and repetitive) to access the data and instead lets you just say what data you want and where do you want it.

Second, by utilizing cloud services Endev removes certain configuration requirements (like installing a database to store your data) and overhead (programming a web server just so you can read APIs that don’t support cross-origin resource sharing).

So how does that look like? Let’s find out…

About the tutorial setup

Throughout the tutorial you will be shown examples of code and will be encouraged to play with it as well. The code will be presented in CodePen, an interactive tool for working with snippets of HTML. You can choose to stay inside CodePen and play there. You can freely modify the examples (by selecting Edit on Codepen in upper left corner of every example) and click Run to see the modifications. If you want to save your modifications you need to click Fork. You don’t need to even have an account to do that, but if you want to be able to later find your code easily it’s suggested that you do create an account.

If instead you want to work outside CodePen that’s easy as well. Endev does not require any special installations. The only thing you need is a text editor (not Word, but Notepad or TextEdit or similar, I would recommend Sublime) and a browser. You can then create a text file and save it with html extension (e.g. “example01.html”). Finally, add the following code to your file:

<!DOCTYPE html>

<html lang="en">

<head>

   <meta charset="UTF-8">

   <title>Endev Tutorial</title>

   <link rel="stylesheet" href="https://maxcdn.bootstrapcdn.com/bootstrap/3.3.6/css/bootstrap.min.css">

   <script src="www.endevjs.org/endev.full.js"></script>

   <script src="www.endevjs.org/endev-codepen-tutorial.js"></script>

</head>

<body>

   <!-- CODE FROM THE EXAMPLES COMES HERE -->

</body>

</html>

And then you just copy the code from the examples to the dedicated location. To test the code just open your file in the browser.

Good luck!

Explain the example

Let’s understand better what the new annotations represent:

  • with data-value annotation we have declared a reference label itemName for the value entered into the input box. This allows us to reference that value in other places (as we will see soon).
  • with data-click annotation we declare what should happen when the button is clicked. In this case insert({name:itemName}) specifies that a new object should be inserted into a collection and that its name should be the value from the input box (referenced by itemName). But what collection does it get inserted in?
  • well that’s what insert-into="WishList" is for. This annotations specifies the collection where the data from the "insert(...)" operation is supposed to go. And this is our desired WishList collection.
  • finally, we replaced the two static list items (<li>..</li>) we had before with one annotated with a from annotation where we define the query to get the data. This way we have defined a template that will be duplicated for each element in the retrieved data. In our case we said WishList item meaning we want to get the data from WishList and item label to reference the individual item (similar to how we used itemName earlier). We can see how the item is used inside the {{item.name}} where we just want to display the item’s name. Double curly brackets are used whenever we want to just display data we have somehow retrieved or created through annotations.

Now if you have any experience with web you may think: “This is a nice trick, but if I close my browser the data will be lost”. Well, go ahead and try . Still there? If you opened the same browser your data should still be there. The catch is in “the same browser”. Endev use browser storage to dynamically create your data structure and store your data. It is similar to how cookies work and that is why the data is only local and in another browser or another computer you would not see it. But we will solve this issue soon.

2. Real-time applications and data in the cloud

In the previous chapter we have seen how to create a simple interactive application with real data that gets created on-the-fly. All of this was achieved using the Endev JavaScript library, but without writing a single line of JavaScript code. Instead of writing the code to tell our application how to get the data, we used annotations to declare what data we want and where we want it. Endev took care of the rest.

Now let’s see what else can Endev do for us. In fact, in all of our examples from the previous chapter there was an important feature that we didn’t discuss. We already know that the data is stored in the browser and if we restart it, our data should not be lost. But what happens if you open this tutorial in two tabs of your browser and edit the data in one of them? Well, in fact, the data will be instantly synced to the other tab as well. Go ahead and try this on the last example from previous chapter:

See the Pen Endev Tutorial: 1.5 by Filip Kis (@filipkis) on CodePen.

This is a so called real-time synchronization technique and apps that support it are called real-time apps with most famous example being Google Docs.

2.1 Putting the data into the cloud

To put this into the context of our WishList application, creating a list just for ourselves is not so much fun (who will then get us all those nice things). Our goal from the start was to be able to share the list with others. And real-time support ensures that as soon as somebody makes a change to the list, others can see it right away. However, we have a big limitation in our current app - the data is stored only in the browser. So to share it, somebody would actually need to use our computer and that’s, of course, not what we had in mind.

Here is where Endev’s second core principle comes to play. Remember what was the second principle? Cloud services. By using a cloud service called Firebase, Endev taps into the power of real data storage that can be accessed from anywhere. And since it is a cloud service, it doesn’t require complex setup and maintenance. In fact, for this tutorial you will need no setup at all.

Note: Since now the data is stored in the cloud, the data is shared with anybody taking this tutorial at the same time. Just so that you are not surprised if new things start appearing and disappearing from the list :)

Let’s see how we can “turn on” this feature in Endev. Here is our example from before, but now running with Firebase:

See the Pen Endev Tutorial: 2.1 by Filip Kis (@filipkis) on CodePen.

Explain the example

It looks almost the same. The only difference is that where we used WishList before now we have firebase:WishList. With this change we have told Endev that we do not want to anymore take the information from local browser storage, but instead from Firebase. In fact, the previous annotation we used for local storage was just a shortcut version, the full version would be local:WishList. We can already start to see a pattern here of how we define different services (providers). In the next chapter we will see two more services that we can use to access the data.

As with local storage, Endev will create the WishList (or any other name you put there) collection in Firebase if one didn’t exist before.

Time to make our WishList app shareable. The interface we have built so far is for managing the wish list, however we do not want our friends or family to start adding or removing the items on the list. So let’s create a new interface where they will only be able to see the list and choose what they want to get (before you do that, make sure you have some items in the list if you deleted them in previous step):

See the Pen Endev Tutorial: 2.2 by Filip Kis (@filipkis) on CodePen.

Explain the example

This example shows already familiar from and data-click with update(..) annotations. However, there are two new things to notice here.

First, a new annotation data-if is used to make an element appear only when a specific condition is fulfilled. We wanted “[reserved]” to appear next to the items that somebody has already registered for.

Second, we are using the update statement in a new way. The first argument (item), as before, tells us which object will be updated, but the second argument specifies what information to update. This is useful when we do not have input boxes or other similar controls but we still want to change some values. We could, for instance, say update(item, { registered:true, when: $now() }) where we would also set item’s when property to know when somebody registered. $now() is a special function in Endev that gives you the current date and time.

2.3 Showing the list of items user registerd for

Now that the users can register for an item, it would be nice that they also know what they registred for. Remember, now that we use Firebase service, everybody will see the same list. So it might happen that somebody forgets what they decided to get. One way we could do this is having people sign up with their name (and maybe other information), but let’s look at different way so we can explore another feature of Endev - possibility to combine data from different services.

To achieve our goal, we can store the items that a user has registered for in the local browser storage. That way each user will see only their own list (of course, only if looking with the same browser, but for prototyping purpose we can be satisfied with that).

2.4 Hiding taken items from the main list

Now, instead of showing “reserved” next to the item that’s been taken in the wish list, we could consider only showing the items that are not yet taken. In other words, we want to filter out the wish list:

See the Pen Endev Tutorial: 2.4 by Filip Kis (@filipkis) on CodePen.

Explain the example

To filter any data, we use where annotation in combination with from. The statement of the where annotation specifies the condition. In our case the condition is item.registered != true. If we want to specify several conditions we can separate them with AND. For instance, if we would have a price property of the item on the list and wanted to add also a condition that the price must be less than 100 the statement would look like item.registered != true AND item.price < 100.

2.5 Allowing users to remove themselves from an item

As a final task, let’s add the remove button so the users can change their mind. Naturally, the item should then appear back in the list as not registered:

See the Pen Endev Tutorial: 2.5 by Filip Kis (@filipkis) on CodePen.

Explain the example

The button needs to do two things: it needs to update the item in the WishList to make it not registered anymore and delete the item from the ToGetList. Both of these actions are described with the, by now, familiar data-click annotation. However we needed extra from and where annotations on the button. The reason is that the update statement needed a reference to which item in the WishList to update. If we just wrote update(getItem, ...) the item in the ToGetList would be updated. If, on the other hand, we wrote update(item, …),  where item refers to  the one from the list we used at the beginning of the code, that wouldn’t do anything. Endev references that are defined by from annotation are only available inside the element where they are defined. Therefore, we needed to use a new from annotation and limit it, with the where annotation, to those items that are the same as the item in our ToGetList.

This is what we call nested queries. In other words, queries where results from one query is used to effect the other query. The insert-into part in this example is another such query, albeit a more simpler one.

Summary of what we have learned in this chapter

  • Endev supports real-time applications, in other words, data in your interface is kept in sync between different users/instances of your application
  • You can use Firebase service to have the data stored in the cloud and accessible from anywhere
  • where annotation in combination with from allows us to filter the results of the query
  • You can combine data coming from different data sources. For instance, adding data from one source to another or using the data from one source to filter the data in the other one. When doing so you will get the so-called nested queries
  • The data-if annotation can be used to show and hide elements depending on the value assigned to it

3. Accessing API data

In the previous chapters we have worked with data we created ourselves. We either created and stored it locally, in the browser, or using a service to store data in the cloud so it can be accessed from anywhere. Normally applications that we design will have some own data, as in our examples, but there is also plenty of data already out there that can be used and combined to build useful, creative and fun applications.

There is a variety of issues concerned when working with API data. From finding the right data for your needs, understanding it, getting access to it (not all data is open or easily accessible), using the right technology to read it; all the way to increasingly popular field for data science that aims at developing complex algorithm combined with scientific analysis to gain new knowledge from it. It is for sure an interesting domain and even though you can probably find “an app” for anything these days, there are still plenty of possibilities for creating new or improving existing applications.

3.1 Accessing the data from REST API

In this chapter we will work with APIs and touch upon some of the mentioned issues and see how Endev combines various methods to resolve them. To start with let’s look at a simple example of two APIs that are accessed through Endev annotations:

See the Pen Endev Tutorial: 3.1 by Filip Kis (@filipkis) on CodePen.

Try and see what happens when you add explain="lottery" to the element that uses from to query the lottery results.

Explain the example

Both examples use REST APIs which are basically web links but instead of going to a nice webpage they display raw data. The raw data can come in different formats, but in recent years JSON format (what you see for example if you look at http://apis.is/lottery) became most dominant.

If you click on the titles of the two APIs, they will lead you to the documentation where you can see what are all the API endpoints (other name for the API web addresses) and parameters (the values you see at the end web address, typically after “?” mark). On the Open Movie Database page we can see that there is only one endpoint (http://www.omdbapi.com/?) and there are several parameters that we can send. You can also test your inputs on their website, but the results are not very readable.

Let’s get back to our example. In the first case we are using the from annotation to access the movie API. As we before specified firebase:WishList and local:ToGetLis, now we use the web address to specify that we want to access the API. The address must include the “http:” or “https:” as that’s how Endev knows it is a REST API. We can already see some parameters (i.e. ?plot=short&r=json) defined in the web address. When accessing REST APIs you can choose whether to keep the parameters in the from annotation or define them in the where statement. Apart from where statement being more readable, more importantly, it gives us a familiar way of defining conditions that link to other parts of the interface. In this example, we compare the t property to the value in the input field represented by title reference. As soon as we type something in the input box, the new API data will be retrieved that satisfies the given conditions.

In the second example we have a much simpler API that displays the results of Icelanding Lotto draws. The lottery API gives us data where the actual data of interest nested under the results property. Therefore, we need to use nested queries to access it. Luckily the documentation website contains the nice example where we see how the data looks like. However, as we saw with the previous example, that is not always the case. In such situations there is an explain annotation that can be placed next to the from annotation such as explain="lottery". Go and try it out and see what happens? The annotation temporary replaces anything that’s inside the element annotated with from and instead just displays the full data retrieved from the API. You can use it for testing and exploring the data.

3.2 Using YQL service for uniformed access to supported APIs

Working with API endpoint addresses tedious: finding the right address, knowing what parameters to use, different syntaxes and results formats, etc.  It is time to bring out the fourth and final service provider (remember we had local storage, Firebase and REST so far) - Yahoo Query Language (YQL). YQL is an online service that can be seen as a proxy to many other data sources.

YQL uses what they call “data tables” to define different data sources. The tables have certain parameters (some optional, other mandatory) that can be set through where conditions. Based on the table you specify and the conditions you set in your request, YQL knows how to transform that information into a call to the actual web service and then it reads and returns the results.

To explore what data sources YQL supports we can use YQL Console. There you can see the list of supported tables on the left side by clicking on them an example of how to use them. The initial list contains only the tables that relate to Yahoo service (flicker, Yahoo Weather, etc.), but if you select “Show Community Tables” you can get more than 1000 tables that allow you to access many other services including popular ones like Twitter, Google, etc. And all of those table and their data can be accessed through Endev.

Let’s look at an example that uses YQL to access the public photos on from Flickr API. Furthermore, the example lets you save the photos to your private collection (using local store). The example contains minimal styling (the class attributes) to make it better fit on the page:

See the Pen Endev Tutorial: 3.2 by Filip Kis (@filipkis) on CodePen.

Try adding describe="flickr.photos.search" to the element with similar from and see what happens.

Explain the example

By now you probably know to first look for from annotation as that is where we always start from. It looks similar to how we used it with local storage and Firebase, i.e. we start with stating the service and the data table we want to use yql:flickr.photos.search followed by result - the future reference label of our choosing that will hold the results. In the where statement we specify the attributes that the API needs. If we are not sure what parameters can be used, we can use describe="flickr.photos.search" annotation. Go ahead, try it out. It works similar to how we used explain earlier to see all the raw data, just instead it show the possible parameters. This annotation works only with YQL service. It will not give us the meaning of the parameters, but it will link to API’s documentation (when known to YQL) where more information on the parameters can be found.

The second from annotation is an interesting new case. Flickr API returns a result that is not directly a collection of images. Instead it returns an object that has property called photo where the collection of photos matching our search will be present. Therefore, we need a second from annotation that access that collection. This time we don’t specify any service because Endev already knows where result comes from. Often data we work such nested collections and we will need to use nested queries to access them.

Another interesting case with Flick API is that it does not give us exact address of the image, but instead gives use pieces of information (server, server farm, id, etc.) that we then combine into a full address in the src attribute of image element to show the photo.

The image tag also contains already familiar annotations for inserting the image into PhotoCollection local storage. We have, however, decided not to store the only the parts, but the full address in our private collection, therefore we combine it again when passing it into the insert statement.

Finally, we list the photos from our PhotoCollection.

3.3 Accessing any API with YQL (and avoiding CORS problems)

When working on the client side code, as we’ve been working so far, there is a limitation to accessing certain APIs in the form of Cross-Origin Resource Sharing (CORS). CORS poses certain security risk and for those reasons it’s normally not enabled, which in turn prevents us from accessing the data from the browser code. What we would need in such case is a server to read the needed data and send it back to us. However, because it’s turned off by default, there are many open APIs that have no security concerns (they share the open data) but still are not accessible in this way. A small example is this API that gives the location of the International Space Station. Try reading it with Endev .
